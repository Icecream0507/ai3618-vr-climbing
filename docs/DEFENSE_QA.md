# 答辩 Q&A —— Summit VR（平衡 + 落脚点 VR 攀岩）

> 给主讲 + 各模块负责人。每题给「一句话先答 + 展开 + 如果追问」。
> 课程 AI3618。核心卖点：**在只有头 + 两手柄的 VR 上，把"身体平衡 + 落脚点"做成可玩机制**，而市面攀岩 VR 几乎全是"纯手、没脚、没平衡"。

---

## 一、立意 / 选题

**Q1：为什么做攀岩？又为什么强调"平衡和落脚点"，而不是像别的 VR 攀岩那样抓着爬就行？**
- 一句话：现有 VR 攀岩（The Climb、Gorilla Tag 类）可玩性几乎只剩"手抓-拉"，把真实攀岩里**最核心的身体平衡和脚法**全丢了；我们想把这一层补回来。
- 展开：真实抱石（bouldering）的难点常常不在手有多大力，而在**重心怎么落在脚上**、要不要换脚、会不会"开门"（barn-door）甩出去。我们把这件事做成机制：探出支撑范围就失衡掉，踩对脚 / 换对侧手就稳住。
- 追问"凭什么说别人没做"：见 `docs/RESEARCH.md` 的市场调研——主流产品都是手部交互；学术上 Kosmalla(CHI 2020) 明确指出 **VR 攀岩里脚比手更影响沉浸感**，但很少有产品落地脚法。

**Q2：你们的创新点到底是什么？**
- 一句话：用**头显当重心(CoM)** + **抽象虚拟脚**，在三点式 VR（无脚追踪）上低成本地实现了平衡 + 落脚机制——这是个"可运行的存在性证明"。
- 三个支点：① 平衡模型（头当重心，判断它有没有落在支撑点的横向范围内）；② 落脚点（脚自动吸附到下方落脚握点，拓宽支撑面）；③ 二者都是**加法层**，关掉就退化成普通手抓攀岩，保证基线一定能玩。

---

## 二、平衡模型（技术核心，最可能被追问）

**Q3：平衡是怎么算的？是物理仿真吗？**
- 一句话：**不是**物理引擎仿真，是一个轻量的几何判据——把头(HMD)当重心，看它的横向位置有没有落在"所有接触点（手 + 脚）的横向跨度"之内。
- 展开：接触点投影到墙面横向轴上得到一个区间 `[min,max]`，再放宽一点容差 `supportMargin`；重心在区间内 → 稳（分数 0~1，越居中越稳），在区间外 → 失衡（0~-1，越远越惨）。失衡持续超过 `graceTime` 才开始扣平衡条，扣空就"脱落"。
- 为什么不用真物理：作业口径是"能玩就行、别过度工程"；真做 ZMP / 支撑多边形 / 逆运动学既不稳又调不动，几何判据**可调、可解释、跑得动**，而且足够还原"开门"的体感。代码在 `ClimbMath.StabilityScore`，纯函数、有 9 条单测。

**Q4：用头当重心，合理吗？头又不是重心。**
- 一句话：是简化，但有**文献支撑**且对"探身失衡"这一类最关键的失败模式足够准。
- 展开：参考 Mitsuda & Kimura(Frontiers in VR 2026) 的"头作为重心代理 + 重心越过支撑脚即判定跌倒"模型（n=24 实验）。攀岩时人贴墙、躯干基本竖直，头的横向位置和重心高度相关，用它判"有没有探出支撑"误差可接受。这是**工程取舍**，不是物理真值——我们在报告里明确写了是经验近似。
- 追问"误差多大/怎么验证"：诚实回答——目前靠仿真自检验证了**机制正确性**（失衡→脱落→重生→登顶闭环 10/10），**手感/参数的人因验证**是列在评测计划里的待办（NASA-TLX / 跌落原因统计），见 Q11。

**Q5：那些常量（supportMargin、drain、grace 之类）怎么定的？**
- 一句话：经验初值 + 留了 Inspector 接口随时调，不是从物理推导。
- 展开：`supportMargin=0.06m`、`maxOvershoot=0.30m`、`drain=0.6/s`、`regen=0.7/s`、`grace=0.35s`。grace 是防抖（手部追踪抖动不该立刻扣血），drain<regen 让"扶正就能回血"。全部是 tuning 常量，文档里如实标注"by feel"。

## 三、落脚点 / 脚法

**Q6：没有脚部追踪（Quest 就头+两手柄），脚是怎么来的？**
- 一句话：**抽象虚拟脚**——身体下方的落脚握点自动吸附成两只"脚"，给平衡模型提供额外支撑点，而不是真去追玩家的脚。
- 展开：从头往下估一个"落脚区"，左右各找最近的 Foot/Either 握点吸住；爬过去后脱离再吸下一个（foot-gluing）。脚是**从握点派生的状态**，不是 IK 腿。

**Q7：为什么不用 IK 反解腿 / Meta 的生成式腿？**
- 一句话：那些在攀岩这种极端姿势下会崩，且与我们机制无关。
- 展开：IK 腿在贴墙、高抬腿、侧身时极易穿模/抖动，调试成本高、收益低；我们要的只是"脚在哪、撑不撑得住"这个**信息**，抽象状态就够，还便于可视化（脚下橙色标记）。这条也有文献建议（保持脚抽象优于硬上 IK）。

## 四、玩法 / 工程

**Q8：移动是怎么实现的？会不会晕？**
- 一句话：**反向位移**——手抓住握点后，身体朝手的反方向移动，等于"把自己拉上墙"，和 Unity XRI Climb Provider / The Climb 同一原理。
- 展开：抓着的手每帧的位移，反向施加到角色身上（`ClimbMath.ClimbDelta`），握点在世界里保持不动。最近抓的手主导，松开交还另一只手。晕动方面：攀岩是手主动驱动的自我位移，比摇杆滑行舒适；真上头显可再加 vignette（列在可选项里）。

**Q9：线路是怎么做的？有几条？**
- 一句话：程序化生成，**4 条**预设线路，无需美术也能跑。
- 展开：`RouteBuilder` 用基本几何体搭墙 + 彩色握点 + 顶部登顶触发器。握点颜色：黄=手、橙=脚、紫=任意、绿=登顶、红=易碎、蓝=休息。4 条：Warm-up / Balance Test（同侧连续动作逼你用脚维持平衡）/ The Arete（易碎点逼你别久留）/ Endurance（体力管理 + 伪装休息点的易碎诱饵）。

**Q10：你们说"测试通过"，是怎么测的？没头显怎么测？**
- 一句话：写了个**脚本机器人**驱动真实游戏逻辑做端到端自检，外加纯数学单测，全自动、无需头显。
- 展开：`SimulatedClimber` 像真玩家一样抓握、探身、爬墙，断言"失衡会脱落、坠落会重生、登顶会计时、脚会落点"等 10 条；`ClimbMathSelfTest` 9 条验证平衡数学。命令行 `HeadlessCheck.Run` 一键跑，出退出码 + 报告。**当前 9/9 + 10/10 PASS**。
- 加分点（可主动说）：这个机器人在跑的时候**抓到一个真 bug**——`CharacterController.minMoveDistance` 默认 0.001m 会吞掉慢速攀爬的每帧位移导致"爬不动"，纯看代码发现不了，只有实跑才暴露，已修复。说明我们不是"写完就交"，是真在引擎里验证过。

## 五、评测 / 局限 / 未来

**Q11：有没有做用户实验 / 量化评测？**
- 诚实版：目前完成的是**系统 + 仪器 + 协议**，机制正确性已用仿真闭环验证；**人因数据（找 5–8 人玩，记录完成时间 / 按原因分类的跌落次数 / NASA-TLX 工作负荷 / SSQ 晕动）是已设计好、待执行**的下一步，模板在 `REPORT.md`。不谎称已有数据。
- 话术："我们把'能不能测'这件事先做扎实了——系统可玩、指标可采、协议写好了；课程时间内先保证机制跑通，评测是明确的 next step。"

**Q12：最大的局限是什么？**
- 头当重心是近似，竖直贴墙时好、大动态/倒挂时会失真；脚是抽象的、没有真实力反馈；平衡参数还没经过人因调校。这些我们都在报告里如实写了，没有夸大。

**Q13：为什么不接真头显演示，而是放仿真视频？**
- 一句话：作业口径是"仿真验证即可"；我们把重心放在**机制本身**而非外设接入。
- 展开：核心贡献（平衡 + 脚法）在仿真里能完整、可复现地展示，且不受手追踪噪声干扰，反而更清楚。XR 接入是纯工程接线、不影响机制成立，留作可选扩展。

**Q14：这东西真比"纯手攀岩"更好玩 / 更像真攀岩吗？**
- 诚实版：这正是我们想用 Q11 的评测去回答的问题，现在还不能下结论。但设计上它**强制玩家像真攀岩一样思考重心和脚**——同侧连续动作不踩脚就会被甩下来——这在纯手攀岩里是没有的。

## 六、分工 / 课程

**Q15：五个人各做了什么？**
